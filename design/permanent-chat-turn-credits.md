## 落地状态（已实现代码）

- 表字段：`usr_users.turn_balance`（Prisma `User.turnBalance`）
- 并存规则：有效档位 **无** 日/月限额 → 扣 `turn_balance`；**有** 日历限额 → 走原日/月 + `token_balance` 兜底
- free 种子改为无日/月限额；种子启动时会清掉已有 free 的日历限额，并为 `turn_balance=0` 的 free 用户回填注册赠送次数
- 系统变量：`quota.signup_turn_grant`（默认 `"20"`）
- 错误码：`QUOTA_TURNS`
- 管理端用户页可编辑余额 / 增量加次

详见 changelog `003-permanent-chat-turn-credits.md`。

# 永久对话次数额度 vs 按月计量：并存分析

## 结论（先看）

| 问题 | 答案 |
|------|------|
| 能否实现「永久次数」？ | 能。语义清晰：余额制，不过期，用一次扣一次。 |
| 与现有日/月限额能否并存？ | 能。二者计量单位都是「对话轮次」，但**重置规则不同**，必须在扣费优先级上显式约定。 |
| 是否必须改表？ | **建议改表**（至少给用户加一个「永久轮次余额」字段）。不改表也能勉强 hack，但会污染现有语义、难以运营。 |
| 能否复用 `token_balance`？ | **不建议**。它是 LLM `total_tokens` 量纲，不是「对话次数」。 |

---

## 一、现状：两套已并存的额度机制

当前 `manager`（Prisma）与 `server`（GORM 同库）已经是**双轨**：

### A. 会员档位 · 日历重置轮次

- 表：`sys_membership_tiers`（`daily_limit` / `monthly_limit`）
- 用量：`usr_user_usage.usage_count`（按 UTC 自然日聚合；月额度 = 本月 SUM）
- 行为（`server/internal/authz/authz.go` → `EnsureChatQuota`）：
  - 日限额满 → 硬拦截 `QUOTA_DAILY`（**不可**用 token 绕过）
  - 月限额未满 → 放行，本回合走「包含轮次」
  - 月限额已满且 `token_balance > 0` → 仍可聊，事后按 LLM token 扣钱包
  - 月限额已满且余额为 0 → `QUOTA_TOKENS`
- 种子：`free` 10/日·100/月；`plus`/`pro` 更高
- **自然月一过，已用轮次「自动清零」**（其实是新月份计数从 0 开始，无独立 reset 任务）

### B. 永久 Token 钱包

- 字段：`usr_users.token_balance`
- 来源：商店 `consumable` 商品的 `token_grant`
- 消耗：仅在「月包含轮次用尽后」按 LLM `total_tokens` 扣减
- **不过期，用完即无** —— 这点与「永久次数」很像，但**单位是 token，不是次数**

### 管理端现状

- 可改档位日/月限额、用户 `tierId`、某日 `usageCount`
- **不能**在 UI 里直接编辑 `token_balance` / 订阅到期
- **没有**「给指定用户加 N 次对话」的一等能力

---

## 二、你想要的「永久次数」是什么

典型产品语义：

1. 新用户初始赠送 **20 次**对话机会（总量，不按月刷新）
2. 运营可对指定用户 **追加 N 次**（同样不过期）
3. 每成功完成一次可计费对话 → 余额 **-1**
4. 余额为 0 → 不可再聊（除非另有订阅/月包权益）

这与现有「free 档每月 100 次」是**对立重置模型**：

| 维度 | 按月计量（现状 A） | 永久次数（目标） |
|------|-------------------|------------------|
| 上限来源 | 档位 `monthly_limit` | 用户个人余额 |
| 时间 | 自然月重置 | 永不因时间重置 |
| 运营加量 | 改档位或改当日 usage | 给个人余额加 N |
| 用尽后 | 可进 token 钱包 | 需另定规则 |

因此：**不能**简单把 free 的 `monthly_limit` 改成 20 来冒充永久额度——下个月又会回到 20。

---

## 三、能否与按月规则并存？

**可以并存**，前提是先定「扣费/放行优先级」。推荐一种清晰、可落地的规则：

### 推荐策略：订阅月包优先，永久次数作补充钱包

对每一次即将发生的对话轮次：

```
1. 若用户有有效订阅档（plus/pro 且未过期）
   → 走现有日/月限额逻辑（含 token_balance 兜底）
   → 不扣永久次数

2. 否则（free / 订阅已过期）
   → 若永久次数余额 > 0：放行，成功后余额 -1
   → 否则：拒绝（新错误码如 QUOTA_TURNS）

3. 日限额（可选）
   → 若仍要对永久次数用户做「防刷」：可保留 daily_limit
   → 若永久次数本身就是稀缺资源：可对 free 把 daily 设为 0（不限日）或不检查日限额
```

这样：

- **付费订阅用户**：体验与现在完全一致（按月计量 + token 兜底）
- **免费/运营赠送用户**：走永久次数，不会每月自动「回血」
- 两种规则在同一用户生命周期内可切换（买订阅 → 用月包；到期 → 回到永久次数余额）

### 不推荐的并存方式

- **同时扣两边**（月计数 + 永久次数各 -1）：用户难理解，运营难算账。
- **永久次数仅当月额度满后才扣**：语义变成「第二种 token 钱包」，但单位又是次数，和 `token_balance` 叠床架屋。
- **把永久次数塞进 `token_balance`**：单位冲突；内购发放、LLM 扣费、管理端展示全部要改语义，风险高。

## 四、表结构要不要改？

### 方案 0：不改表（不推荐）

| Hack | 问题 |
|------|------|
| 把 free 的 `monthly_limit` 调成 20 | 每月重置，不是永久 |
| 改历史 `usage_count` 假装没用过 | 只能「减已用」，不能「加永久额度」；跨月失效 |
| 滥用 `token_balance`，约定「1 token = 1 次」 | 与现有 LLM token 扣费、内购 `token_grant` 冲突 |

结论：产品要「永久次数 + 运营加次」，**实质需要用户级余额字段（或等价账本）**。

### 方案 1：最小改表（推荐起步）

在 `usr_users` 增加：

```text
turn_balance  Int  NOT NULL DEFAULT 0   -- 永久对话次数余额
```

可选配套：

```text
-- 系统变量或 free 档种子：注册时初始赠送次数（如 20）
-- 不必进档位表；进系统设置更灵活
```

**改动面：**

| 层 | 改动 |
|----|------|
| Prisma / 迁移 | `User.turnBalance`（由你方执行 migrate） |
| Go entity / repo | 读写、原子 `+N` / `-1` |
| `EnsureChatQuota` | 按第三节优先级分支 |
| 对话成功路径 | 在「走永久次数」时 `-1`（与记 `usage_count` 可并存：usage 仍做统计） |
| Manager 用户页 | 展示余额 + 「增加次数」操作 |
| `/users/me` | 返回 `turn_balance` |

`usr_user_usage` **不必为额度再改结构**——继续当「用量事实表」即可；永久额度的「还能用几次」以 `turn_balance` 为准。

### 方案 2：余额 + 流水账本（运营更稳）

额外表例如 `usr_turn_credit_ledger`：

- `user_id`, `delta`, `reason`（`signup_grant` / `admin_grant` / `chat_consume` / …）
- `operator_id`（管理端加次时）
- `created_at`

余额仍可用 `turn_balance` 冗余字段（写操作同事务更新），或由流水 SUM 得出（查询成本更高）。

适合：要审计「谁给谁加了多少次」、客诉对账。

### 方案 3：商品化（与商店打通时）

若以后要卖「次数包」：

- `BillingProduct` 增加 `kind=turn_pack` 或 `turn_grant` 字段
- 核销时 `AddTurnBalance`，与现有 `token_grant` 对称

初期若仅后台加次，可先不做。

## 五、与现有 free 档如何衔接

当前 free = 10/日 · 100/月（每月刷新）。若上线永久次数，需要产品二选一（或分阶段）：

1. **替换 free 月包**  
   - free 档：`daily_limit`/`monthly_limit` 置 `0` 或 `null`（表示不走日历限额）  
   - 注册写入 `turn_balance = 20`  
   - 免费用户只认永久次数  

2. **双轨过渡**  
   - 老用户继续月包；新用户走永久次数（用系统开关 / 用户标记）  
   - 复杂度更高，一般不值得  

3. **并存但分人群**  
   - 有订阅 → 月包  
   - 无订阅 → 永久次数  
   - free 档的 `monthly_limit` 对「无订阅」不再生效  

推荐产品落地：**选项 1 + 第三节优先级**（订阅用户仍用 plus/pro 月限额）。

## 六、实现时注意点（manager / server）

1. **扣减时机**：与现网一致——AI 成功回复后再扣；失败不扣。  
2. **并发**：`UPDATE ... SET turn_balance = turn_balance - 1 WHERE turn_balance > 0`，避免超卖。  
3. **日限额**：永久次数模式下是否仍检查 `daily_limit` 需产品拍板。  
4. **统计**：继续写 `usage_count`，便于报表；不要用 usage 反推永久余额。  
5. **缓存**：`LoadPrincipal` 已缓存 `token_balance`，需把 `turn_balance` 一并纳入并在加减后 `InvalidatePrincipal`。  
6. **错误码**：新增 `QUOTA_TURNS`，客户端文案与 `QUOTA_TOKENS` 区分。  
7. **Prisma 迁移**：按项目约定由开发者本地执行生成，不在自动化流程里代生成脚本。

---

## 七、建议决策清单

落地前只需拍板这几项：

1. 永久次数与月包的优先级是否采用「有订阅用月包，否则用永久次数」？  
2. free 档是否取消日历月限额，改为注册赠送 `turn_balance`？  
3. 初始赠送次数（如 20）放系统变量还是写死种子？  
4. 管理端是否只要「加减余额」，还是要流水账本？  
5. 永久次数用尽后，是否仍允许用现有 `token_balance` 继续聊？（建议：**否**，避免两套永久钱包并存；token 钱包留给订阅超月包场景。）

---

## 八、总答

- **能实现**永久额度控制；与按月计量是两种规则，但可通过「订阅走月包 / 非订阅走永久次数」清晰并存。  
- **建议改表**：`usr_users.turn_balance`（可选 ledger）；**不要**复用 `token_balance`。  
- `UserUsage` 可继续只做统计；额度真相在用户余额 + 档位限额。  
- 主要改动在 `server` 的 `EnsureChatQuota` 扣费链，以及 `manager` 用户管理加次 UI；billing 可第二期再接次数包。
